# AGENTS.md

This file provides guidance to Codex (Codex.ai/code) when working with code in this repository.

## Project

CueBud is a native macOS 15+ SwiftUI app that provides real-time communication coaching during video calls via a floating overlay. All audio/video processing is on-device (Apple Speech, Apple Vision, AVFoundation).

## Commands

The Xcode project is generated by XcodeGen from `project.yml` — it is not checked in. Regenerate whenever `project.yml` changes or dependencies shift.

```bash
# Regenerate CueBud.xcodeproj (required after editing project.yml)
xcodegen generate

# Build + run all tests
xcodebuild test -scheme CueBud -destination 'platform=macOS'

# Run a single test (class or method)
xcodebuild test -scheme CueBud -destination 'platform=macOS' \
  -only-testing:CueBudTests/TipEngineTests
xcodebuild test -scheme CueBud -destination 'platform=macOS' \
  -only-testing:CueBudTests/TipEngineTests/testCooldownBlocksRepeatedTips

# Build only
xcodebuild build -scheme CueBud -destination 'platform=macOS'
```

Open `CueBud.xcodeproj` and press Cmd+R to run. On first launch the app shows a Google sign-in screen; after auth the onboarding flow walks through microphone, camera, and speech-recognition permission grants.

## Architecture

### App launch flow

```
CueBudApp
  └─ auth.currentUser == nil  →  LoginView
  └─ !hasCompletedOnboarding  →  OnboardingView
  └─ else                     →  OverlayView (coaching session)
```

`AuthService` (`Services/AuthService.swift`) is owned by `CueBudApp` as a `@StateObject` and is **independent of the coaching pipeline**. It performs a Google OAuth PKCE flow, exchanges the code for tokens, upserts the user into Supabase (`users` table), and stores the refresh token in the Keychain via `KeychainHelper` (`Utilities/KeychainHelper.swift`). On every launch it silently refreshes the session in the background (`refreshSessionInBackground()`). `handleCallbackURL(_:)` must remain wired to `.onOpenURL` in `CueBudApp` — that is the only place macOS routes the OAuth custom-scheme callback back into the app.

User profile (`AuthUser`) is cached in `UserDefaults` (`cuebud.currentUser`) so the first render skips `LoginView` for returning users without a network round-trip. `signOut()` clears both the Keychain refresh token and the UserDefaults cache.

`SettingsView` receives `AuthService` as an `@EnvironmentObject` (injected in `CueBudApp`) and shows account info plus a Sign Out button.

### Coaching pipeline

The coaching layer is a one-way reactive pipeline from **raw sensors → analyzed snapshots → coaching decisions → UI**, wired with Combine `PassthroughSubject`s. Understanding these seams is essential before modifying anything non-trivial:

```
AudioAnalysisService ──segmentSubject/partialTranscriptSubject──▶ SpeechCoach ──tipSubject──┐
                                                                                            ├──▶ TipEngine ──displayTipSubject──▶ OverlayViewModel ──▶ OverlayView
VideoAnalysisService ──snapshotSubject──▶ PostureCoach ──tipSubject────────────────────────┘
                                                         SessionViewModel also consumes segmentSubject/snapshotSubject for aggregate SessionMetrics
```

Key invariants:

- **`SessionViewModel`** (`ViewModels/SessionViewModel.swift`) owns every service and coach as `let` properties and is the single source of truth for session lifecycle. `startSession()`/`stopSession()` must start/stop both services, the tip engine, and reset both coaches — adding a new coach means wiring it in both places plus `setupTipEngineSubscriptions()`.

- **`TipEngine`** (`Services/TipEngine.swift`) is the only place tip display policy lives. Coaches are deliberately dumb: they emit a `CoachingTip` every time a condition is observed. The engine handles warmup (15s), cooldown-after-dismiss (90s per `TipType`), priority-queued concurrent tips, dedup, and a grace-timer-based auto-dismiss. **Do not add cooldown or rate-limiting logic inside coaches** — that breaks the re-emit/grace-timer contract that keeps persistent tips visible while the condition is still active.

- **`AudioAnalysisService`** (`Services/AudioAnalysisService.swift`) recycles the `SFSpeechRecognizer` session every 55s because Apple caps recognition at ~60s. The recycle finalizes the current transcript as a `SpeechSegment` and starts a fresh `SFSpeechAudioBufferRecognitionRequest`. Any change that touches this loop must preserve the recycle, and `requiresOnDeviceRecognition = true` is non-negotiable (privacy is a product requirement — audio never leaves the device).

- **`VideoAnalysisService`** (`Services/VideoAnalysisService.swift`) runs Vision requests on a background `processingQueue` at ~3 fps (throttled via `minProcessInterval`). Frame analysis is `nonisolated` and hops to `@MainActor` only to publish the resulting `PostureSnapshot`. Keep Vision work off the main actor.

- **Two-tier speech detection**: `partialTranscriptSubject` (real-time, throttled to 1s in `SpeechCoach`) drives live feedback; `segmentSubject` (emitted on session recycle/finalization) drives accurate aggregate counts in `SessionViewModel`. Both are needed — don't collapse one into the other.

- **`OverlayViewModel`** observes coaches and `TipEngine` to populate the overlay; the overlay `NSWindow` is reconfigured in `CueBudApp.configureOverlayWindow()` to float above all apps including fullscreen (`.canJoinAllSpaces`, `.fullScreenAuxiliary`).

All `@MainActor`-isolated types (services, coaches, view models) must only be mutated on the main actor. When adding async work, hop back via `Task { @MainActor in ... }` before touching `@Published` state — the existing code follows this pattern consistently.

## Tests

Tests live in `CueBudTests/` and are pure unit tests (no capture session, no speech recognizer). They target the deterministic policy layer: `FillerWordDetector`, `SpeechCoach` thresholds, `PostureCoach` snapshot logic, and `TipEngine` cooldown/priority/grace-timer behavior. When adding a new coaching rule, add a corresponding test at this layer rather than trying to test through the live audio/video services.
